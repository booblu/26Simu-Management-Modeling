# ⚙️ 07. 自动化流水线与一键验收提交流程

## 📌 导言：为什么你的分数不由人类决定？

在传统的作业收集中，学生最常说的一句话是：“老师，在我的电脑上明明能跑通的，是你的环境配错了。”
在真实的软件工程和高维量化风控领域，这句话等同于渎职。

为了模拟最高标准的工业界协作，本课程的每一次作业（Sprint）交付，**必须是一套全自动的数字机器**。导师和评阅机器人**绝不会**手动去 `pip install` 你缺少的奇怪依赖，也**绝不会**在一堆乱七八糟的名字里寻找我们要的那个画图脚本。

只有完全跑通自动化流水线（CI）的仓库，才具备被“人工批阅（读你的 Memo 报告）”的资格。如果机器报红，即使你的策略再聪明，得分也是 0。

## 📂 1. 绝对纪律：标准化的空间拓扑 (The Repository Structure)

你的项目大本营必须严格按照下述骨架陈列。多一个少一个字都不行，这是机器法官找证据的路线图：

```text
Student_Workspace/
  ├── spec.yaml                # 【灵魂】你的假设边界、核心参数与 V&V 测试声明
  ├── requirements.txt         # 【地基】你代码的唯一依赖清单，只能包含本课允许的白名单库
  ├── src/
  │   └── simulator.py         # 【引擎】所有的 V3底层数学、Agent策略，全塞这里。不要把代码全堆在外面！
  ├── tests/
  │   └── test_invariants.py   # 【红线】使用 pytest 编写的系统守恒极压测试
  ├── experiments/
  │   ├── scenarios.json       # 【输入剧本】极端震荡、单边崩盘的数据配方配置
  │   └── run_all.py           # 【点火器】一键执行文件！导师只敲这一行代码。
  ├── results/
  │   ├── metrics.csv          # 【弹道日志】跑完 run_all 自动生成的干扁数据表
  │   └── figures/             # 【战报】跑完 run_all 自动生成的策略对比高清折线图
  └── Decision_Memo.md         # 【帅令】你作为 CEO 给出的最终 1页纸 决策报告
```

## 🚀 2. 导师的“一键点火”协议 (The 1-Click Execution)

在每个 Demo Day，评阅机器人（以及导师本人）会将你的仓库拉取（Clone）下来，然后在你的根目录下，**只会执行底下这一段且只有这一段死命令**：

```bash
# 1. 机器人根据你的地基配置环境
pip install -r requirements.txt

# 2. 机器人调用法官测谎仪，如果报红（Failed），当场处决，游戏结束。
pytest tests/ -v

# 3. 机器法官按下点火按钮，等待不超过 60 秒的极速渲染。
python experiments/run_all.py

# 4. 机器人去检查 results 目录下是否如约吐出了 metrics 数据和 figures 图表。
# 定向抓取你的 Memo 进行最后的人工或 LLM 审查。
```

**你的任务：** 就是用尽一切手段，保证上述这四步在任何一台干净的云主机上，都能像德芙巧克力一样丝滑。

## 🤖 3. 终极自证：在云端唤醒你的 CI (GitHub Actions)

为了不让你在截稿日因为“本地能跑，云端挂了”而抱头痛哭，你需要自己在仓库里架设一个 CI（持续集成）安检门。

这才是本课程中“一阶码农”与“二阶管理架构师”的核心区别：**让自动化来为你打工。**

1. **什么是 CI？**
   在你的 `.github/workflows/ci.yml` 里，存放着一段机器指令。每次当你敲下 `git push` 把代码推送到云端时，GitHub 的官方服务器会自动分配一台干净的虚拟机给你。
2. **它会干什么？**
   这台虚拟机会自动重复上面第 2 节的“一键点火”协议。如果成功，你的 GitHub 仓库主页会亮起一枚神圣的**绿色对勾 (✔)**。如果失败，亮起**红叉 (❌)**。
3. **得分规则**：
   **任何带有 ❌ 的 commit，被视为“未交付的垃圾模型”**，我们拒绝阅卷。只要你的红叉一天没修成绿灯，迟交扣分法杖就会一直砍你的分数。
