# 🛠️ 05. 全栈工具链与开源依赖基准规范 (The Toolchain)

## 📌 导言：为什么我们需要“书同文，车同轨”？

在管理建模与仿真的世界里，最可怕的噩梦不是模型跑不出结果，而是**你的模型换一台电脑就跑不起来了**，或者你用的商业闭源仿真软件（如 AnyLogic 等）把底层的随机数发生器封装成了你根本看不见的黑盒。

为了确保你在本课程（以及未来你的创业生涯中）交付的决策模型具备**绝对的可重现性 (Reproducibility)、完全的透明度 (Transparency) 以及 100% 的自动化验收能力**，我们制定了本基准规范。

在接下来的 4 个 Sprint 中，你的系统必须，且只能基于以下公共开源基础工具链进行构建。任何人不得引入未经导师（或 CI 法官）批准的第三方私有框架或重量级非主流包。

## 🐍 1. 核心计算与仿真引擎底层 (Core Engine)

基于未来 AI Agent 对代码生成的极度友好性，以及生态系统的丰富度，我们抛弃了传统商学院的专有仿真器，全线拥抱纯 Python 栈。

- **基础设施**：`Python 3.10+` (不可低)。这是你所有一切的地基。
- **环境隔离**：不允许全局乱装包。你的每一次提交必须自带 `requirements.txt` 或使用 `poetry`/`venv`。
- **高精度数学核芯（破除 Decimal 幻觉）**：
  > ⚠️ 不要把“向投资人展示的价格（如 2500 USDT/ETH）”和“智能合约底层跑的物理状态（如 sqrtPriceX96）”混为一谈。
  
  由于 Uniswap V3 中含有极致变态的 `sqrtPriceX96` 与小数位累计，传统的商科思维倾向于用 Python 的 `decimal.Decimal` 来处理。**这是一个致命的架构级错误！** 以太坊虚拟机 (EVM) 是基于 256 位整数 (`uint256`) 和按位截断法则运行的。如果你的底层引擎使用基数为 10 的浮点数算法，你将产生巨大的精度漂移，在套利者眼中这就是提款机。
  - **强制核心**：在处理池子底层计账（Reserves）、计价（sqrtPrice）或交叉 Tick 手续费扣留时，**必须且只能使用 Python 自带的原生无限精度整数（BigInt：`int`）**，配合按位运算（如 `<< 96` 与 `//`）进行缩放计算。这能在 Python 里 1:1 像素级复现 Solidity 的物理法则。
  - **Decimal 的合法使用区**：仅当你需要生成 `metrics.csv`、画图或者向 CEO 汇报时，才能将内部算好的大整数除以 $2^{96}$ 后，交由 `decimal.Decimal` 格式化为人类可读的价格。

## ⚙️ 2. 大一统专用仿真框架：唯一指定 `Mesa`

在 8 周的高强度交付中，要求你们同时学习排队论框架（SimPy）、复杂网络（NetworkX）和多主体（Mesa）是违背管理学“聚焦”原则的。

因此，为了降低工具带来的认知摩擦，**本课程的四大 Project，强制采用 `Mesa` 作为唯一贯穿始终的仿真调度框架。**

为什么要用 `Mesa` 统一天下？因为不论是单体协议，还是复杂市场，本质都是“实体（Agent）在环境或时间（Model/Scheduler）中产生交互”：

- **在 P1 中**：你只需借用 `Mesa` 的 `Model` 基类来保存 Uniswap V3 的池子状态，借用 `DataCollector` 来极其优雅地收集每一步的数据。此时还没有 Agent，只有环境。
- **在 P2 中**：你（作为唯一的 LP）变成了第一个注册进 `Mesa` 调度器（`BaseScheduler`）的 Agent，随着时间步的推进，你对这个池子发出调仓动作。
- **在 P3 中**：协议变成了多个环境变量，而你的 Agent 变成了负责跨协议调度的超级管理员。
- **在 P4 中**：`Mesa` 展现真正威力。你将向 `RandomActivation` 调度器中塞入 1000 个散户 Agent、50 个毒性套利 Agent。你不用再写任何 `while` 与 `for` 循环，`Mesa` 会自动帮你公平或随机地调度这场成千上万人参与的黑暗森林战役。

> **一句话总结：** 底层数学核（`Decimal`）负责算钱，系统级大一统框架（`Mesa`）负责推进时间、调度大兵团（Agents）和全自动收集面板数据（DataCollector）。严禁在这门课中引入 `SimPy` 或其它仿真框架。

### ⚠️ Mesa 实战避坑与生存指南 (Survival Guide in Mesa)
在构建复杂的 DeFi 金融模型时，`Mesa` 也有其极度危险的局限性。你必须把以下三条军规刻在脑子里，否则在后期的极压测试中必死无疑：

1. **单线程算力墙 (Performance Wall)**：`Mesa` 是纯 Python 写的，极度易读但也极度缓慢。当你在 P4 中尝试塞入几十万个 Agent 时，内存会瞬间被写爆。
   - **解法**：系统建模看重的是“涌现”，不是无限堆人头。通过 `Mesa` 强大的 `batch_run` 进行多进程参数寻优，并将单局仿真里的 Agent 数量克制在数百个的核心代表即可。
2. **拒绝随机调度，拥抱 MEV (The MemPool Override)**：`Mesa` 自带的 `RandomActivation` 会让所有套利者掷骰子决定谁先出牌。这在流行病学里是管用的，但在加密货币黑暗森林里，交易是按 Gas 费高低被矿工打包的（这就是 MEV 抢跑）。
   - **解法**：到了 P2 之后，你必须**重写或者继承** `BaseScheduler`，打造一个带有拍卖排序功能的池子（MemPool Scheduler）。谁出的手续费高，你让谁优先在 `step()` 里被执行。
3. **不要让 DataCollector 吞噬高频日志 (High-Frequency Logs)**：`Mesa` 的 `DataCollector` 是以“回合 (Step)”为单位收集全盘快照的。如果一个 Step 内部发生了 10 次疯狂的来回砸盘又被拉回原位，`DataCollector` 是看不见的（降采样遗漏）。
   - **解法**：算账的归算账。在你的 `V3_Pool` 核心合约代码内部，手动埋点高频 `event_log`，而绝不仅仅依赖外层的 Mesa 回合收集器。

## 🚦 3. 门禁法官与极压测试台 (Verification & Validation)

你不应该靠人眼去盯终端的日志，你需要机器替你 24 小时站岗。

- **日常断言法官 (The Daily Judge)**：`pytest`
  - 这是本课程绝对的核心。你的 `tests/` 文件夹下必须布满用 `pytest` 标注的红线。它能极其优雅地输出错误发生时的局部变量状态（Traceback），是证明你的系统“未被污染”的第一防线。
- **混沌猴子与模糊测试 (The Chaos Monkey)**：`Hypothesis` (推荐用于高强度 V&V)
  - **基于属性的测试（Property-based Testing）**：如果你只测试 `swap_x(100)`，这意味着你是在“硬编码”猜答案。但 `Hypothesis` 能帮你在一秒钟内自动生成 10 万个带有恶意的、分布在各种极其刁钻边界（如无穷大、负数、NaN、0.00000001）的输入数据去疯狂攻击你的协议模型。只要它没击穿红线，你就可以安心地告诉投资人你的系统是鲁棒的。

## 📊 4. 弹道记录与决策画像 (Data Analytics & Visualization)

当你的协议仿真循环跑完 100 万个 Step 后，所有的轨迹数据需要被结构化并提炼成“CEO 的望远镜”。

- **数据提纯与矩阵**：`pandas` + `numpy`
  - 严禁打印（`print`）十万行日志交差。你的 `experiments/` 目录下的主函数必须最终生成一份极其干净、行列清晰的 `metrics.csv` 或 `.parquet` 数据集。`pandas` 是这部分唯一的官方指定语言。
- **商业战术画像**：`matplotlib` 和 `seaborn`
  - 拒绝丑陋的默认蓝线图。你的图表是给决策层看的。必须使用这两者来绘制**带误差带（Error bands）的收益曲线**、**参数敏感性的热力图（Heatmap）**，以及**策略崩盘的散点图**。所有的出图脚本必须能够离线无交互运行（即代码敲下，图表自动存入 `results/figures/` 文件夹，而不能指望弹出窗口让你手动去点保存）。

## 🤖 5. 上下文与 Agent 连通器 (The AI Workflow Hooks)
*（注：此部分在 Sprint 3 与 Sprint 4 进入深水区时启用）*

当我们开始让多个 Agent 在你的池子里博弈（如：套利狗 vs 痴呆散户），你的 Python 代码必须成为可以被 LLM 调用的引擎。

- **配置文件规范**：所有影响你环境风暴强度的参数必须剥离为纯文本：`spec.yaml` 或 `config.json`。
- 这是为了在最后一步，让你可以接驳例如 `LangChain` 或 `AutoGen` 等外部的 Agent 层。这些外层大模型脑子只要改写这一个 `.yaml` 里的参数配置项，就能驱动底层的纯 Python 环境不断重新运行，从而实现“AI 自动调参并寻找最优解”的神迹。
